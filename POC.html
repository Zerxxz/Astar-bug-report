Astar DApp Staking — Builder Social Links & Validator.isURL Bypass

📋 Bug Overview

Summary: The DApp Builder registration form accepts and stores arbitrary javascript: protocol URLs as social account links. Because validator.isURL() in Validators.ts does not restrict allowed protocols, payloads like javascript:alert(document.domain) pass validation. When rendered in the public DApp page (Builders.vue), the URL is placed directly into an <a href> attribute, causing persistent XSS on every visitor.

🔍 Technical Details

1. Vulnerable Validator (Validators.ts)
import validator from 'validator';

// ❌ VULNERABLE — No protocol whitelist
export const isUrlValid = (url: string): boolean =>
  (url ? validator.isURL(url) : false);

Without protocols: ['http', 'https'] and require_protocol: true, validator.isURL() accepts any protocol including javascript:, data:, and vbscript:. This is documented behavior.

2. Vulnerable Sink (Builders.vue — PUBLIC PAGE)
<!-- Social links rendered directly into <a href> without sanitization -->
<a :href="team.githubAccountUrl" target="_blank" rel="noopener noreferrer">
  <img src="github-icon" />
</a>

<a :href="team.twitterAccountUrl" target="_blank" rel="noopener noreferrer">
  <img src="twitter-icon" />
</a>

All team builder social URLs are rendered as-is in href attributes on the public DApp page. Anyone visiting the DApp page becomes a victim.

⚠️ Note on Vue Security

Vue's default template escaping does NOT protect against XSS in attribute values. When data is bound via :href or v-bind:href, Vue inserts the raw string value — including javascript: — into the DOM. The browser then parses <a href="javascript:..."> and executes the JS on click.

✅ Recommended Fix

Fix 1: Strong URL Validation (Validators.ts)
import validator from 'validator';

// ✅ SECURE — Protocol whitelist + require protocol
export const isUrlValid = (url: string): boolean =>
  url ? validator.isURL(url, {
    protocols: ['http', 'https'],
    require_protocol: true,
    require_tld: true,
  }) : false;
Fix 2: URL Sanitization Before Render (Builders.vue)
import DOMPurify from 'dompurify';

// Sanitize URL before binding to href
const safeUrl = DOMPurify.sanitize(team.twitterAccountUrl, {
  ALLOWED_URI_REGEXP: /^(?:(?:https?|mailto):|[^a-z]|[a-z+.\-]+(?:[^a-z+.\-:]|$))/i
});

// Safe href binding
<a :href="safeUrl" target="_blank" ...>
Fix 3: Content Security Policy (CSP) Header
# server config (nginx/apache)
Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline';
# Note: 'unsafe-inline' needed for inline Vue styles; prioritize Fix 1 & 2
